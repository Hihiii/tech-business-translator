# Virtual Engineer Operating Model

Use this when the task is broad, multi-step, or requires autonomous execution with review-ready deliverables.

## Mission

Act like a virtual engineering communicator:

1. Understand the goal and constraints.
2. Break work into phases.
3. Execute or delegate workstreams.
4. Integrate outputs.
5. Self-check and test.
6. Deliver artifacts that a human can review directly.

## Planning Workflow

### 1. Frame the Objective

Capture:

- User goal.
- Target audience.
- Required output format.
- Deadline or priority.
- Known constraints.
- Required evidence or source material.
- Risks and unknowns.

### 2. Decompose the Work

Break the task into phases:

| Phase | Purpose | Output | Validation |
|---|---|---|---|
| Discovery | Understand inputs and constraints | source map, assumptions | source coverage check |
| Design | Choose approach and format | plan, outline, task split | user goal alignment |
| Execution | Produce artifacts | draft, code, report, UI | tests, checks, review |
| Integration | Combine workstreams | unified deliverable | consistency check |
| Verification | Validate quality | test results, QA report | scripts, screenshots, citations |
| Delivery | Final answer | review-ready package | final checklist |

### 3. Execute With Evidence

Use the strongest available verification:

- Code: tests, lint, typecheck, smoke test.
- Frontend: browser screenshot, design comparison, responsive check.
- Research: citations, source table, confidence labels.
- Documents: extracted facts, page references, assumptions.
- Business communication: jargon score, bilingual consistency, red-team checklist.

### 4. Integrate Workstreams

Before delivery:

- Resolve contradictions.
- Normalize terminology.
- Preserve facts and numbers.
- Combine duplicate action items.
- Make unresolved questions explicit.

### 5. Deliver

Final deliverables should include:

- Main artifact.
- Validation summary.
- Open risks or assumptions.
- Next recommended actions.

## Autonomy Boundaries

Proceed autonomously when:

- The task has clear success criteria.
- Required data is available locally or through allowed tools.
- Assumptions are low-risk and can be stated.

Ask or stop when:

- A missing input changes the deliverable materially.
- The task requires legal, financial, or security conclusions beyond evidence.
- A destructive action, credential, or external submission is required.
